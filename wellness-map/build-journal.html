Build journal and rebuild reference

How we built ABCD x Wellbeing Atlas

This document records how the atlas was shaped through a back-and-forth collaboration: the user brought planning questions, usability concerns, and civic intent; Codex translated those into data choices, interface changes, QA checks, and rebuildable implementation steps.

Open atlas Benchmark details Data QA README
What it is A static Singapore neighbourhood planning lens for access, HDB affordability, transit, parks, and climate comfort around a postal code or map point.
Why it exists To help people spot neighbourhood strengths and gaps, compare places fairly, and start better planning conversations from shared evidence.
How to treat it Use it as a comparative planning signal. It is not a certified site assessment, live routing engine, or measured block-level climate model.

The Collaboration Pattern

The build did not start as a fixed specification. It evolved like a product workshop. The user repeatedly tested the live map, noticed where the interface or assumptions felt wrong, and asked practical questions such as whether a postal code really matched the map, why the counts looked fixed, whether colours were enough, what "p" meant, and how the tool could become useful for planning rather than just lookup.

Codex responded by inspecting the app, changing the code, testing locally, explaining tradeoffs, and refining the next layer. This created a loop: build, observe, question, decide, verify, then improve.

Team decision style: We favoured transparent approximations over false precision. Where a dataset was incomplete, licensed, or not exact enough, the app labels it as a proxy, marks provenance, or points to future data work.

Chronology Of Major Decisions

1. Start with care

We began with the question of whether Singapore has a comprehensive health and healthcare facilities database. The answer became a practical product idea: combine available official/open datasets into one map rather than waiting for a perfect single database.

2. Broaden wellbeing

The user asked to add public sports and exercise facilities, parks, and a postal-code search. The scope shifted from "health map" to "wellbeing atlas". Distance from a searched postal code or dropped map point became the main interaction.

3. Fix location trust

Early versions had postal-code and map-position mismatches. We moved towards exact OneMap postal-code geocoding where possible and made the selected point explicit so users can trust what is being measured.

4. Make all records visible

The user noticed that headline counts were not fully reflected in lists and map markers. We tightened the rendering logic so counts, visible layers, radius filters, side lists, and map markers stayed aligned.

5. Add hover feedback

The user wanted popups on map hover, not only when clicking the sidebar. We added marker popups so the map itself became more inspectable.

6. Replace reviews

Google reviews would require API credentials, licensing compliance, and ongoing cost. We replaced them with quality signals that the static app can honestly support: official source, access information, and distance.

7. Add age groups

The atlas expanded to ageing facilities, children facilities, and then separate youth facilities after the user correctly pointed out that teenagers may not appreciate being lumped with children.

8. Improve accessibility

Colours alone became hard to distinguish, so we added shapes to markers and layer toggles. This made categories more legible for more users and at dense zoom levels.

9. Turn lookup into planning

The user asked whether each postal code is above or below average against its constituency, nearby constituencies, and Singapore. We added benchmark modes, percentiles, counts within 500m and 1km, and a details page with examples ranked above and below.

10. Add affordability

HDB resale prices and HDB median rents were added so access could be read alongside affordability. We decided higher price or rent should be treated as pressure in the visual chips.

11. Add transit

MRT stations, bus terminals/interchanges, and future Circle Line Stage 6 reminders were added. We included nearest transit chips and distances in the map overlay.

12. Add climate visuals

The user wanted sun, wind, and heat information without clutter. We added a toggleable climate lens with animated sun path, wind flow, PM west sun, and a heat proxy. Later, we made PM west sun and heat proxy month-sensitive.

13. Tame the interface

As the sidebar grew, the user asked to avoid one long scroll. We converted lower sections into expandable panels and added a short upfront description of what the tool is most useful for.

Product Principles We Settled On

Planning lens, not verdict

The app should help users ask better questions. It should not pretend to be the final authority on a neighbourhood.

Show the basis for comparison

Percentiles and "above/below" labels only make sense when users can see what they are being compared against.

Prefer official/open signals

When Google reviews were impractical, official source, access type, provenance, distance, and QA notes became the trust layer.

Visual first, text when needed

Dense planning information should be scannable through chips, shapes, colours, toggles, and map overlays before users read details.

Separate demographics respectfully

Children, youth, and ageing facilities are related, but they serve different populations and should be visible as distinct layers.

Label proxies clearly

Population-weighting, heat exposure, and transit distances are useful proxies, but the app should say what they are and what they are not.

Interface Decisions

Map and search

Postal-code search and dropped map points are the core entry points. The selected point is shown explicitly so users know what the distances and benchmarks refer to. The app uses Leaflet with OpenStreetMap tiles for a static, GitHub-friendly MVP.

Markers and popups

Markers use both colour and shape. This was a direct response to the user noticing that colour alone was not visually clear enough. Popups appear on hover so the map can be explored without requiring sidebar clicks.

Counts and benchmarks

The sidebar shows counts at 500m, 1km, and the selected radius. Benchmark cards compare the selected point against its constituency, nearby constituencies, and Singapore. Percentile was changed from compact "p" notation to clearer explanatory text where needed.

Map overlay

The expandable overlay on the map summarises HDB affordability, access, transit, and city-centre proximity as visual chips. This was added so planning signals can be read on the map itself, not only in the left panel.

Climate lens

The climate layer is off by default. When opened, it zooms to a useful planning scale, shows animated sun and wind visuals, and restores the previous map view when closed. PM west sun rotates with the month, the sun path has simple roughly three-hour time tags, and heat proxy responds to greenery, centrality, PM sun, and wind season.

Fengshui Fun Lens

A compact playful card in the expanded map overlay translates existing signals into Water, Green, Flow, Heat, and Balance bars. It is deliberately labelled fun, not formal, and does not add a separate map layer or claim to be a traditional assessment.

Sidebar structure

After the app accumulated many useful sections, lower panels became collapsible: benchmark scores, housing, nearest by category, nearest overall, and facility list. This keeps the first screen focused while preserving depth.

Benchmark Logic

The benchmark system compares a selected point with residential benchmark points rather than arbitrary map space. This matters because planning questions are usually about lived access.

Residential locations Each HDB residential building or postal point is treated equally. This is simple and transparent.
Population weighted HDB footprint area is used as an open-data proxy weight until dwelling-unit or population-weighted data is added.
Percentile interpretation: A percentile of 80 means the selected point ranks above about 80 percent of comparison points for that metric. For access and transit, higher is generally better. For HDB resale and rent, higher is treated as affordability pressure.

Rebuild Guide

Use this sequence if a future developer or AI needs to recreate the atlas from scratch.

  1. Create a static web app with `index.html`, `styles.css`, `app.js`, and a `data/` folder of JavaScript data bundles.
  2. Use Leaflet and OpenStreetMap tiles for the map so the app can run from GitHub Pages without a backend.
  3. Define facility categories first: healthcare, ageing, children, youth, sport, park, and connector.
  4. Normalize each record into a shared shape: name, category, subcategory, latitude, longitude, source, access/sector, official signal, and optional source URL.
  5. Geocode postal-code based datasets through OneMap and cache results locally in the data build process.
  6. Render map markers from the same filtered record list used for counts and sidebars, so counts and pins cannot drift apart.
  7. Add hover popups directly to markers, and keep category shapes distinct from colours.
  8. Build benchmark points from HDB residential locations and constituencies from electoral boundaries.
  9. For each category and radius, count facilities around every benchmark point and compute averages and percentiles for constituency, nearby constituencies, and Singapore.
  10. Create a separate benchmark details page that lists examples just above and below the selected point.
  11. Aggregate HDB resale and rental data by town and flat type over the selected period, then infer the selected town from searched address text when possible.
  12. Add transit nodes and compute nearest MRT, bus terminal/interchange, and future station distances.
  13. Add a climate lens only after the core access/affordability map works. Keep sun, PM west sun, wind, and heat proxy visually distinct.
  14. Add QA and methodology pages early. They become more important as the atlas gains layers.
  15. Keep every proxy labelled. Do not hide uncertainty just to make the interface feel more complete.

Known Limitations And Future Upgrades

Not live routing Not measured UHI Not full private market data Static MVP Open-data first
  • Walking time, cycling time, and public transport travel time should eventually replace straight-line distances where routing data is available.
  • Climate and heat should be upgraded with measured or modelled land-surface temperature, shade, tree canopy, building mass, and wind corridor data.
  • Future transit stations should be reviewed when official opening dates and coordinates change.
  • Population-weighted benchmarks should move from footprint proxy to dwelling-unit or population-weighted points.
  • Private residential sale and rental data should be added only if licensing and source terms allow it.
  • A server-side geocoding layer would avoid exposing API keys and improve exact search coverage.

What Future Builders Should Preserve

Preserve the spirit of the collaboration: keep the app legible, honest, and planning-oriented. When a user asks "what does this mean?", the interface should already be halfway to answering. When a dataset is imperfect, mark the limitation rather than burying it. When two groups have different needs, give them separate layers. When the map gets too busy, use toggles, collapsible panels, and visual summaries rather than more text.

Core sentence to keep: A planning lens for spotting neighbourhood strengths and gaps around a Singapore postal code: access, HDB affordability, transit, parks, and climate comfort. Benchmarks are comparative signals, not certified site assessments.
